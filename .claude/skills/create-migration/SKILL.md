Create a new Supabase migration for this project.

## Steps

1. Ask the user what the migration should do if not already specified
2. Generate the migration file using: `bun run db:create-migration <name>`
   - Name format: snake_case describing the change (e.g., `add_user_email_index`)
3. Write the SQL into the generated file under `supabase/migrations/`
4. Validate the SQL:
   - Ensure idempotent operations where possible (IF NOT EXISTS, IF EXISTS)
   - Include both the migration and a comment header describing the purpose
5. Show the user the final migration file for review

## Naming Convention

Existing migrations follow this pattern:
```
YYYYMMDDHHMMSS_description.sql
```
The timestamp is auto-generated by `supabase migration new`.

## Reminders

- After creating: run `bun run db:migrate` to push to Supabase, or `bun run db:migrate:local` for local reset
- This is a single-user KST (UTC+9) application — no multi-tenant considerations needed
